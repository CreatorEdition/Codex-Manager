# 2026-06-06 网关模型校验移除全局路由自举

## 状态

✅ 已完成

## 背景

网关请求进入模型校验时，旧逻辑会先按执行计划触发模型路由 bootstrap：

- 账号池路由会遍历账号来源并维护账号模型映射。
- 聚合 API 路由会遍历聚合 API 来源并维护聚合模型映射。
- 混合路由会同时触发两类全局维护。

在几千账号或大量聚合 API 场景下，这会把普通推理请求变成潜在的全局 DB 读写和路由维护任务，放大 CPU、WAL 写入和请求延迟。

## 修改

- `model_route_error()` 不再在请求热路径调用全局模型路由 bootstrap。
- 删除仅供请求路径使用的 `bootstrap_model_routes_for_plan()`。
- 保留现有模型目录、API Key 模型组权限、显式映射校验。
- 无显式映射时，仅通过已发现的 `model_source_models` 判断当前路由类型是否有可用上游来源。

## 验证

- 更新网关模型校验测试，确认聚合 API 路由可接受已发现 source model。
- 测试同时断言请求路径不会自动写入 `model_source_mappings`。

## 后续

- 模型路由维护应继续由模型管理页、来源同步和显式刷新流程承担。
- 若需要自动关联新发现模型，应放入后台维护或管理端显式操作，不能回退到每次网关请求执行全局 bootstrap。
